# Requirements Document

## Introduction

This document specifies the requirements for implementing a secure API key-based authentication system that connects Minecraft plugin servers to the RaindropCentral backend platform. The system enables server owners to register their servers, authenticate via time-limited API keys, and establish persistent connections that link all server data (players, inventories, statistics) to the authenticated server entity.

## Glossary

- **RCentralServer**: The database entity representing a registered Minecraft server in the RaindropCentral platform
- **API Key**: A cryptographically secure token generated by the backend that authenticates a server
- **Server Owner**: The Minecraft player (identified by UUID) who creates and owns a server registration
- **Backend**: The RaindropCentral web platform that manages server registrations and API keys
- **Plugin**: The Minecraft server-side plugin (RCore) that connects to the backend
- **Server UUID**: A unique identifier for each Minecraft server instance
- **API Key Hash**: A bcrypt-hashed version of the API key stored in the database
- **Heartbeat**: Periodic status updates sent from the plugin to the backend to maintain connection state
- **RServer**: Legacy entity reference that needs to be replaced with RCentralServer
- **RPlayerInventory**: Entity storing player inventory snapshots scoped to a specific server
- **RPlayer**: Entity representing a player profile with statistics and server history
- **Connection Status**: Enumeration tracking whether a server is CONNECTED, DISCONNECTED, or ERROR

## Requirements

### Requirement 1: Server Registration on Backend

**User Story:** As a server owner, I want to create a server registration on the backend platform, so that I can obtain an API key to connect my Minecraft server.

#### Acceptance Criteria

1. WHEN a logged-in user creates a server registration on the backend THEN the system SHALL generate a unique Server UUID
2. WHEN a server registration is created THEN the system SHALL store the creator's Minecraft UUID as the owner
3. WHEN a server registration is created THEN the system SHALL generate a cryptographically secure API key
4. WHEN an API key is generated THEN the system SHALL hash the API key using bcrypt before storage
5. WHEN an API key is generated THEN the system SHALL display the plaintext API key to the user exactly once
6. WHEN an API key is displayed THEN the system SHALL hide the API key after 15 minutes
7. WHEN an API key is hidden THEN the system SHALL never display the plaintext API key again
8. WHEN a server registration is created THEN the RCentralServer entity SHALL have ConnectionStatus set to DISCONNECTED

### Requirement 2: API Key Entry in Plugin

**User Story:** As a server owner, I want to enter my API key in the Minecraft plugin, so that my server can authenticate with the backend.

#### Acceptance Criteria

1. WHEN a server administrator enters an API key via plugin command THEN the Plugin SHALL send the API key to the backend authentication endpoint
2. WHEN the Plugin sends an API key THEN the Plugin SHALL include the Server UUID in the authentication request
3. WHEN the Plugin sends an API key THEN the Plugin SHALL include the Minecraft UUID of the player entering the key
4. WHEN the authentication request is received THEN the Backend SHALL validate the API key against the stored hash
5. WHEN the API key is validated THEN the Backend SHALL verify the requesting player UUID matches the server owner UUID
6. IF the API key is invalid THEN the Backend SHALL return an authentication failure response
7. IF the requesting player UUID does not match the owner UUID THEN the Backend SHALL return an authorization failure response

### Requirement 3: Server Authentication and Connection

**User Story:** As a server owner, I want my server to automatically connect after successful authentication, so that all server data is linked to my registered server.

#### Acceptance Criteria

1. WHEN authentication succeeds THEN the Backend SHALL update the RCentralServer ConnectionStatus to CONNECTED
2. WHEN authentication succeeds THEN the Backend SHALL set the firstConnectedAt timestamp if null
3. WHEN authentication succeeds THEN the Backend SHALL reset the failedHeartbeatCount to zero
4. WHEN authentication succeeds THEN the Backend SHALL return a success response to the Plugin
5. WHEN the Plugin receives a success response THEN the Plugin SHALL store the authenticated server configuration
6. WHEN the Plugin receives a success response THEN the Plugin SHALL begin sending periodic heartbeat messages
7. WHEN a heartbeat is received THEN the Backend SHALL update the lastHeartbeat timestamp
8. WHEN a heartbeat is received THEN the Backend SHALL update server metrics including currentPlayers, maxPlayers, and tps

### Requirement 4: Entity Migration from RServer to RCentralServer

**User Story:** As a system architect, I want to replace all RServer references with RCentralServer, so that the codebase uses a consistent server entity model.

#### Acceptance Criteria

1. WHEN RPlayerInventory entities are created THEN the system SHALL reference RCentralServer instead of RServer
2. WHEN existing code references RServer THEN the system SHALL be refactored to use RCentralServer
3. WHEN the migration is complete THEN the RServer entity SHALL be removed from the codebase
4. WHEN database schemas are updated THEN foreign key constraints SHALL reference r_central_server table
5. WHEN repositories query server data THEN the queries SHALL use RCentralServerRepository

### Requirement 5: Player-Server Association

**User Story:** As a platform administrator, I want player profiles automatically linked to servers they join, so that I can track player activity across the server network.

#### Acceptance Criteria

1. WHEN a player joins a connected server THEN the system SHALL create or retrieve the RPlayer entity
2. WHEN an RPlayer entity is created or retrieved THEN the system SHALL add the RCentralServer to the player's serversJoined set
3. WHEN a player's server association is updated THEN the system SHALL persist the many-to-many relationship
4. WHEN a player leaves a server THEN the system SHALL update the player's lastSeen timestamp
5. WHEN player data is queried THEN the system SHALL efficiently load the associated servers

### Requirement 6: Inventory-Server Association

**User Story:** As a server owner, I want player inventories scoped to my specific server, so that inventory data is isolated between different servers.

#### Acceptance Criteria

1. WHEN a player inventory snapshot is created THEN the system SHALL associate it with the current RCentralServer
2. WHEN a player inventory is restored THEN the system SHALL only restore inventories matching the current RCentralServer
3. WHEN inventory queries are executed THEN the system SHALL filter by both player and server
4. WHEN a server is disconnected THEN the system SHALL preserve all associated inventory snapshots
5. WHEN inventory data is persisted THEN the foreign key SHALL reference the r_central_server table

### Requirement 7: Statistics-Server Association

**User Story:** As a server owner, I want player statistics tracked per server, so that I can analyze player behavior on my specific server.

#### Acceptance Criteria

1. WHEN player statistics are recorded THEN the system SHALL associate them with the current RCentralServer
2. WHEN statistics are queried THEN the system SHALL support filtering by server
3. WHEN a player joins multiple servers THEN the system SHALL maintain separate statistics per server
4. WHEN statistics are aggregated THEN the system SHALL support both per-server and cross-server views

### Requirement 8: Connection State Management

**User Story:** As a platform administrator, I want automatic detection of disconnected servers, so that the system accurately reflects server availability.

#### Acceptance Criteria

1. WHEN a heartbeat is not received within the expected interval THEN the system SHALL increment failedHeartbeatCount
2. WHEN failedHeartbeatCount exceeds the threshold THEN the system SHALL update ConnectionStatus to DISCONNECTED
3. WHEN a server reconnects after disconnection THEN the system SHALL reset failedHeartbeatCount
4. WHEN a server reconnects THEN the system SHALL update ConnectionStatus to CONNECTED
5. WHEN connection errors occur THEN the system SHALL update ConnectionStatus to ERROR
6. WHEN ConnectionStatus changes THEN the system SHALL log the state transition

### Requirement 9: Security and Validation

**User Story:** As a security engineer, I want robust validation of all authentication requests, so that unauthorized servers cannot connect to the platform.

#### Acceptance Criteria

1. WHEN an API key is generated THEN the system SHALL use a cryptographically secure random generator
2. WHEN an API key is stored THEN the system SHALL hash it using bcrypt with appropriate work factor
3. WHEN an authentication request is received THEN the system SHALL validate the API key format
4. WHEN an authentication request is received THEN the system SHALL verify the Server UUID exists
5. WHEN an authentication request is received THEN the system SHALL verify the owner UUID matches
6. WHEN authentication fails THEN the system SHALL log the failure with relevant details
7. WHEN multiple authentication failures occur THEN the system SHALL implement rate limiting
